Аудит-журнал в эпоху LLM: что записывать, чтобы проверка не уничтожила

21 мая 2026 · 4 минуты · аудит, compliance, LLM
Что вы получите за 4 минуты
  • Минимальный набор полей в аудит-таблице, без которых регулятор останется недоволен
  • Почему structlog в файл — это не аудит, а просто логи
  • Готовая схема таблицы audit_events для документной системы с LLM

Почему structlog — это не аудит

Логи в файл — это операционная история. Их можно отротировать, можно потерять при падении диска, можно поменять задним числом, если есть доступ к серверу. Аудит — это набор записей, которые: (а) живут в базе с теми же бэкапами и репликацией, что и бизнес-данные; (б) пишутся только INSERT; (в) переживают удаление документов, к которым относятся. Регулятор спросит «покажите всё, что было с документом X» — у вас должен быть один SQL-запрос с ответом.

Минимум полей

  • occurred_at — TIMESTAMPTZ с индексом DESC.
  • actor — кто действовал. У нас формат: dashboard:<user> для веб-входа, api:<key-hash> для API-вызовов (не хранить ключ, только префикс sha256), worker:<name> для системных событий.
  • action — что произошло. Перечисление с CHECK-ограничением (document.uploaded, document.asked, ...), не свободный текст.
  • document_id с FK ON DELETE SET NULL — чтобы запись пережила удаление документа.
  • request_id — связь с логами и трассировкой.
  • details JSONB — небольшой структурированный контекст (имя файла, обрезок вопроса, тип ошибки). НЕ полное содержимое — это уже в основной БД.

Что специфично для LLM-систем

Каждый вызов модели — это событие, которое нужно записать. Не сам ответ (он часто длинный и потенциально содержит чувствительное), а:

  • Кто спросил (actor).
  • К какому документу относился вопрос.
  • Какой режим поиска использован (keyword / vector / hybrid).
  • Сколько цитат вернулось.
  • Опционально — первые 200 символов вопроса для контекста при разборе жалоб.

В ARA это таблица audit_events с action document.asked. Через год вы сможете ответить на «кто пять раз спрашивал про этот контракт» одним SELECT.

Append-only — как инвариант кода

В коде не должно быть UPDATE audit_events и DELETE FROM audit_events — никогда. На уровне БД желательно отозвать эти права у роли приложения. Но даже без БД-уровня инвариант поддерживается ревью кода: PR с UPDATE/DELETE на этой таблице — это сигнал «надо смотреть очень внимательно».

Не блокирует пользователя

Запись в аудит не должна валить запрос. Если БД временно недоступна — пользователь получает свой ответ; аудит-запись логируется как ошибка для оператора. Лучше пропустить одну запись, чем уронить весь трафик. В ARA это сделано через try/except в AuditService.record().

Что теперь можно сделать
  • Добавьте таблицу audit_events к вашему текущему проекту. Шесть полей, шесть индексов, готово.
  • Пройдитесь по коду и найдите три бизнес-действия, после которых надо звать audit.record().
  • Откройте журнал в демо ARA — там виден живой пример.
