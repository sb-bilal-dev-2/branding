Дизайн API для документного шлюза: 7 решений, которые я бы повторил

15 апреля 2025 · 5 минут · API, архитектура, дизайн
Что вы получите за 5 минут
  • Семь решений, заложенных в API ARA с первого дня
  • Зачем каждое из них стоит лично вам и где оно окупится
  • Ровно два решения, которые я взял бы по-другому, имея опыт сейчас

1. Возврат 202, а не 201 на тяжёлые операции

Загрузка документа запускает OCR и эмбеддинг — это секунды. Шлюз возвращает 202 со ссылкой на ресурс и текущим статусом задачи. Клиент опрашивает статус (HTMX polling раз в 1.5 сек). Никаких висящих HTTP-соединений, никаких таймаутов.

2. Idempotency-Key для всех модифицирующих запросов

Заголовок Idempotency-Key. Один и тот же ключ дважды — один и тот же ответ. Клиенту нужно делать ретраи безопасно; серверу — не плодить дубликаты при сетевых сбоях. Это пять строк кода и спасение от классов багов.

3. Backpressure через 429 + Retry-After

Глубина очереди превысила порог — шлюз отвечает 429 с заголовком Retry-After: 5. Клиент видит, что система устала, и не давит. Это контракт, который позволяет масштабировать вниз без панических хаков.

4. Отдельная схема API-ключа и сессионной авторизации

Веб-дашборд — HTTP Basic. Программный API — X-API-Key. Два разных дерева ответственности: один для человека, второй для интеграции. Не пытайтесь совместить — обмен токенами получится кривым.

5. Версионирование внутри пути, не в заголовке

Когда придёт время v2 — /v2/documents, не Accept-Version: 2. Видно в логах, видно в curl, видно в URL. v1 живёт рядом и обслуживается параллельно.

6. Аудит-журнал сразу, не «потом»

Каждое модифицирующее действие пишет строку в append-only таблицу audit_events: actor, action, document_id, request_id, details. Не «когда потребует регулятор», а с первого дня. Подробнее — в отдельной статье.

7. Один заголовок X-Request-ID для трассировки

Клиент шлёт ID — шлюз использует. Не шлёт — генерирует. ID идёт через очередь во все воркеры (мы кладём его в IngestTask). Каждая лог-строка содержит этот ID. Поддержка становится поиском по одному ключу через все компоненты.

Что я бы изменил

1. Сделал бы лимиты конфигурируемыми per API-key, а не глобальными. Сейчас все клиенты делят один порог; нормальная зрелость — разные тиры.

2. Заранее бы выложил OpenAPI как первоклассный артефакт (с примерами), а не как побочный продукт FastAPI. SDK генерируется из этого; писать интеграцию без него — пытка.

Что теперь можно сделать
  • Откройте API вашей текущей системы и сверьте: какие из 7 уже есть, какие — нет.
  • Самое дешёвое в добавлении — Idempotency-Key и X-Request-ID. Начните с них.
  • Посмотрите OpenAPI в демо ARA по адресу /docs — там видно, как это собирается.
