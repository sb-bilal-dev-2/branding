Зачем государственной системе on-prem LLM вместо облачного API

12 октября 2025 · 4 минуты · LLM, архитектура, безопасность
Что вы получите за 4 минуты
  • Чёткий критерий: когда on-prem LLM нужен, а когда нет
  • Три аргумента в защиту бюджета на свой GPU-сервер
  • Конкретная команда для переключения шлюза с облака на vLLM

Причина 1. Данные не уезжают

Контракты, переписка, кадровые документы не должны отправляться в API чужого облака. On-prem LLM (vLLM, llama.cpp, TGI) держит данные на ваших серверах от запроса до ответа. Это базовое требование любой регулируемой среды — от государственной до банковской.

Причина 2. Стоимость становится предсказуемой

Облачные API считаются токенами. Месяц с 200 пользователями превращается в счёт, который трудно защитить перед финдиректором. Свой сервер — одна капитальная закупка GPU плюс электричество. Видимый, фиксированный, контролируемый. Расчёт амортизации возможен. Расчёт «сколько мы заплатим OpenAI в следующем месяце» — нет.

Причина 3. Контроль над версией модели

Провайдер обновляет модель — ваши промпты, тесты и регрессии ломаются на следующий день. On-prem: вы решаете, когда поднимать новую версию. Можете держать стабильную модель два года, пока не созреет следующая, и переходить по плану, а не по чужому расписанию.

Скрытое преимущество: LoRA под ваши задачи

С vLLM и параметром --enable-lora можно дообучить адаптер на ваших документах (договоры, докладные, регламенты) и подгрузить его рядом с базовой моделью. Адаптер — мегабайты, не гигабайты. Базовая модель остаётся одна; адаптеров может быть несколько под разные задачи.

Это даёт специализацию без затрат на полный fine-tuning. Через облачное API такого контроля нет.

Когда облачный API всё ещё уместен

Не на каждой задаче нужен on-prem. Если вы делаете прототип, и данные не чувствительны, начинать со своего GPU — преждевременная оптимизация. Критерий: данные регулируются (PII, гос. тайна, банковская), и/или объём вызовов делает облачный счёт неприлично большим. До этих порогов — спокойно используйте облако и не торопитесь.

Переключение на vLLM

В ARA смена режима — одна переменная окружения:

ARA_LLM_MODE=vllm
ARA_VLLM_BASE_URL=http://vllm:8000/v1
ARA_VLLM_MODEL=Qwen/Qwen2.5-7B-Instruct

Шлюз сам перенастроит клиента и пойдёт к вашему серверу. Никаких изменений в коде, никаких пересборок образов.

Что теперь можно сделать
  • Откройте свой .env и проверьте, какой ARA_LLM_MODE стоит сейчас.
  • Замерьте текущий месячный счёт за токены — это ваш базовый аргумент перед руководством.
  • Прочитайте как мы делаем аудит-журнал — это естественное продолжение для регулируемой среды.
